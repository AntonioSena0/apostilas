Teoria & Conceitos

A modelagem de dados fundamenta-se na percepção do mundo real como um conjunto de objetos básicos, chamados entidades, e as associações entre eles, chamadas relacionamentos. No processo de abstração do minimundo, é vital distinguir quais objetos possuem vida própria e quais são subordinados a outros para que a estrutura do banco de dados reflita fielmente a realidade observada.

1. Entidade Forte (ou Regular)

Uma entidade é considerada forte quando possui existência independente no ambiente modelado. Isso significa que ela representa um objeto do mundo real (físico como uma pessoa ou um carro, ou conceitual como um curso universitário) que pode ser identificado de forma exclusiva através de seus próprios atributos. Tecnicamente, uma entidade forte possui uma chave primária própria (identificador único) que não depende de nenhum outro objeto do sistema para ser distinguida.

2. Entidade Fraca (ou Dependente)

Diferente da forte, a entidade fraca não possui existência própria individualizada. Sua presença no banco de dados é subordinada à existência de uma entidade forte (chamada de proprietária da identificação) à qual ela está relacionada.

Identificação:

Por definição, a entidade fraca não possui atributos suficientes para formar uma chave primária completa sozinha.

Chave Primária Composta:

Para identificá-la no modelo lógico, utiliza-se a união da chave primária da entidade forte com um conjunto de atributos da própria entidade fraca, chamado de chave parcial (ou discriminador).

Exemplo:

Um Dependente é uma entidade fraca em relação ao Funcionário, pois sua identificação no sistema depende diretamente do registro do funcionário ao qual ele está vinculado.

3. Opcionalidade de Relacionamento (Participação)

A opcionalidade (ou restrição de participação) define se a existência de uma entidade depende de ela estar ou não vinculada a outra através de um relacionamento.

Participação Total (Obrigatória):

Ocorre quando cada instância da entidade deve, obrigatoriamente, estar relacionada a uma instância da outra entidade. É também chamada de dependência existencial.

Participação Parcial (Opcional):

Ocorre quando uma instância da entidade pode existir no banco de dados sem estar vinculada a nenhuma instância da outra entidade.

Exemplo Prático

Abaixo, a implementação física da relação entre Funcionario (Entidade Forte) e Dependente (Entidade Fraca). Note que a entidade fraca herda o numero_seguro_social do funcionário como parte de sua própria identificação e como chave estrangeira.

MySQL

-- Criando a Entidade Forte
  CREATE TABLE Funcionario (
      numero_seguro_social CHAR(9) NOT NULL,
      primeiro_nome VARCHAR(35) NOT NULL,
      salario DECIMAL(10, 2),
      -- Define o número do seguro social como a chave primária
      CONSTRAINT PK_Funcionario PRIMARY KEY (numero_seguro_social)
  );
  
  -- Criando a Entidade Fraca
  CREATE TABLE Dependente (
      -- Atributo herdado da entidade forte (chave estrangeira)
      numero_seguro_social_funcionario CHAR(9) NOT NULL,
      nome_dependente VARCHAR(50) NOT NULL, -- Chave parcial
      sexo CHAR(1),
      -- A Chave Primária é COMPOSTA (PK do Forte + Chave Parcial)
      CONSTRAINT PK_Dependente PRIMARY KEY (numero_seguro_social_funcionario, nome_dependente),
      -- Integridade Referencial com dependência existencial automática
      CONSTRAINT FK_Funcionario_Dependente FOREIGN KEY (numero_seguro_social_funcionario)
          REFERENCES Funcionario(numero_seguro_social)
          ON DELETE CASCADE -- Se o funcionário for excluído, os dependentes também serão
  );

PostgreSQL

-- Criando a Entidade Forte
  CREATE TABLE Funcionario (
      numero_seguro_social CHAR(9) PRIMARY KEY,
      primeiro_nome VARCHAR(35) NOT NULL,
      salario NUMERIC(10, 2)
  );
  
  -- Criando a Entidade Fraca
  CREATE TABLE Dependente (
      numero_seguro_social_funcionario CHAR(9) NOT NULL,
      nome_dependente VARCHAR(50) NOT NULL,
      sexo CHAR(1),
      -- Definição da PK composta e FK com ação em cascata
      PRIMARY KEY (numero_seguro_social_funcionario, nome_dependente),
      FOREIGN KEY (numero_seguro_social_funcionario)
          REFERENCES Funcionario(numero_seguro_social) ON DELETE CASCADE
  );

Dicas & Lembretes

  • Notação de Peter Chen: Em diagramas clássicos de Peter Chen, as entidades fracas são representadas por retângulos de linha dupla e os relacionamentos identificadores por losangos de linha dupla.
  • Chaves Parciais: A chave parcial de uma entidade fraca (como o nome do dependente) é o conjunto de atributos que identifica unicamente as instâncias fracas que pertencem a uma mesma instância forte.
  • Integridade: Ao implementar entidades fracas, utilize sempre a cláusula ON DELETE CASCADE para evitar registros "órfãos" no hardware.
  • Participação Total: Uma entidade fraca sempre terá restrição de participação total em relação ao seu relacionamento de identificação, pois não pode ser identificada sem o proprietário.

Como Aplicar na Prática

  • Itens de Pedido: Em sistemas comerciais, o Item_Pedido é sempre uma entidade fraca vinculada à entidade forte Pedido.
  • Históricos de Salários: Tabelas que registram o Historico_Salarial de um colaborador são dependentes da existência da entidade forte Funcionario.
  • Mapeamento Lógico: Ao converter do modelo conceitual para o relacional, a chave primária da entidade fraca obrigatoriamente deve incluir a chave primária da entidade forte.
