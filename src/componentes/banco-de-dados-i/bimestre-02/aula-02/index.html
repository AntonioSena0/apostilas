Teoria & Conceitos

O autorrelacionamento (ou relacionamento reflexivo) ocorre quando detectamos uma associação entre ocorrências de uma mesma entidade. No mundo real, isso acontece quando objetos de uma mesma classe desempenham papéis diferentes entre si, como um funcionário que gerencia outros funcionários ou um produto que é composto por outros produtos.

1. Conceitos Fundamentais

  • Definição: É um relacionamento binário onde a conexão ocorre com a própria entidade, tratando-a como se fossem dois conjuntos diferentes na abstração.
  • Papéis (Roles ou Alias): No autorrelacionamento, a definição de "papéis" é vital para distinguir o significado de cada participação. Em uma estrutura de peças, por exemplo, uma peça assume o papel de "composta" (o todo) e a outra o papel de "componente" (a parte).
  • Chave Estrangeira Recursiva: Implementa-se através de uma chave estrangeira (FK) que aponta para a chave primária (PK) da própria tabela.

2. Tipos de Conectividade e Aplicações

A. Um-para-Muitos (1:N) - Hierarquia de Níveis

Utilizado para modelar estruturas onde cada item tem apenas um "superior", mas pode ter vários "subordinados".

  • Exemplo (Gerência): Na tabela FUNCIONARIO, o campo nss_supervisor (FK) aponta para o nss (PK) do chefe.
  • Exemplo (Biologia): Uma entidade PARTES_CORPO onde a "Cabeça" é composta por "Olho", e o "Olho" é composto por "Retina".

B. Muitos-para-Muitos (N:N) - Composição (Estrutura de Materiais)

Este é o caso mais complexo, pois um item pode ser, ao mesmo tempo, um "todo" (formado por várias partes) e uma "parte" (integrante de vários outros itens).

Refinando o Exemplo da Fábrica (Explicação Detalhada): Imagine uma única tabela chamada PRODUTO que contém tanto peças simples (parafusos, arruelas) quanto máquinas complexas (motores, carburadores).

  • A Visão do Componente (A Parte): Um "Parafuso" não é exclusivo de um único produto. Ele é usado na montagem de um Bloco de Motor e também na montagem de um Carburador. Ou seja, um único componente participa de muitos produtos compostos.
  • A Visão do Composto (O Todo): Por outro lado, o "Bloco de Motor" não é feito apenas de parafusos. Ele exige Arruelas, Roscas, Pistões e outros produtos. Ou seja, um produto composto exige muitos produtos componentes.

A Resolução Técnica (Entidade Associativa): Como essa relação é N:N, o modelo relacional não permite listar "vários componentes" dentro de uma única linha de produto. Por isso, criamos uma Entidade Associativa (tabela de ligação) chamada COMPOSIÇÃO (ou Estrutura/BOM). Ela terá:

  • ID_Produto_Composto (FK): Referencia o produto que está sendo montado (o "pai").
  • ID_Produto_Componente (FK): Referencia o produto que é usado como peça (o "filho").
  • Atributo de Relacionamento: Geralmente a Quantidade (ex: quantos parafusos vão no bloco?).

3. Comparação: Teoria vs. Prática SQL

Aspecto Teoria (Modelo ER) Prática (SQL)
Identificação Uso de losangos e nomes de papéis. Chave Estrangeira na mesma tabela.
Raiz da Hierarquia O elemento que não possui "pai". Registro onde a FK é NULL.
Consulta Navegação lógica por níveis. Uso de Self Join (Join da tabela com ela mesma via Alias).

4. Demonstração Prática

Implementação da Composição (N:N)

Abaixo, o SQL mostra como a tabela associativa "amarra" a entidade Produto a ela mesma.

-- Entidade Principal
CREATE TABLE Produto (
    codigo_produto INT PRIMARY KEY,
    nome_produto VARCHAR(50) NOT NULL
);

-- Tabela Associativa (A ponte do autorrelacionamento N:N)
CREATE TABLE Composicao (
    id_composto INT NOT NULL,  -- Referência ao "Todo" (ex: Bloco)
    id_componente INT NOT NULL, -- Referência à "Parte" (ex: Parafuso)
    quantidade INT DEFAULT 1,
    
    PRIMARY KEY (id_composto, id_componente),
    FOREIGN KEY (id_composto) REFERENCES Produto(codigo_produto),
    FOREIGN KEY (id_componente) REFERENCES Produto(codigo_produto)
);

O Desafio da Consulta (Self-Join)

Para visualizar quem compõe o quê em uma hierarquia 1:N (como chefes), usamos Aliases (apelidos) para que o banco "finja" que existem duas tabelas separadas.

-- Exemplo de Self Join: Quem é o chefe de quem?
SELECT 
    SUPERVISOR.nome_vendedor AS Chefe, 
    SUBORDINADO.nome_vendedor AS Funcionario
FROM Vendedores AS SUPERVISOR
INNER JOIN Vendedores AS SUBORDINADO 
    ON SUPERVISOR.codigo_vendedor = SUBORDINADO.chefe_fk;

Nesta consulta, a "Tabela A" assume o papel de supervisores e a "Tabela B" o de subordinados.

5. Dicas de Modelagem

  • Opcionalidade: A FK recursiva em hierarquias 1:N deve aceitar NULL. Caso contrário, você nunca conseguiria cadastrar o "Dono da Empresa" (a raiz), pois ele seria obrigado a ter um chefe que ainda não existe.
  • Integridade: Ao excluir um produto que é "composto", a regra de negócio deve decidir se os componentes também somem (Cascade) ou se a operação é bloqueada para evitar dados inconsistentes.
  • Ciclos: Um erro comum é criar um ciclo onde o Produto A é parte de B, e B é parte de A. Isso deve ser evitado via lógica de aplicação para não causar recursividade infinita nas consultas.

Representação Gráfica

1. Modelo Conceitual: Hierarquia 1:N (Exemplo: Funcionários)

Este modelo representa uma estrutura de subordinação simples, onde um funcionário pode gerenciar muitos outros, mas cada subordinado possui apenas um chefe.

Nota: A linha indica que um supervisor pode ter zero ou muitos subordinados, mas um subordinado (opcionalmente) tem um supervisor.

2. Visão de Instâncias: A Árvore de Dados

Para entender como os dados se comportam em uma hierarquia (como partes do corpo ou organogramas), visualizamos a navegação do "topo para as folhas".

graph TD A["Raiz: Diretor (NSS_Supervisor: NULL)"] --> B["Gerente A"] A --> C["Gerente B"] B --> D["Vendedor 1"] B --> E["Vendedor 2"] C --> F["Vendedor 3"] subgraph Legenda L1["FK é NULL na Raiz"] L2["FK aponta para PK do 'Pai'"] end

Conceito: O registro onde a Chave Estrangeira é nula representa o elemento raiz da hierarquia.

3. Modelo Conceitual: Composição N:N (Exemplo: Produtos/BOM)

Utilizado quando um item pode ser composto por várias partes e, ao mesmo tempo, ser um componente de diversos outros produtos (Lista de Materiais - Bill of Materials).

erDiagram PRODUTO ||--o{ COMPOSICAO : "é composto por" PRODUTO ||--o{ COMPOSICAO : "participa como componente em" COMPOSICAO { int id_composto PK, FK int id_componente PK, FK int quantidade }

Resolução: Todo relacionamento reflexivo M:N exige uma entidade associativa (neste caso, COMPOSICAO) para quebrar a relação em duas associações 1:N.

4. Mapeamento Lógico: Composição N:N

Abaixo, a representação de como as chaves são herdadas na tabela associativa para efetivar o relacionamento lógico.

classDiagram class Produto { +int codigo_produto PK +string descricao } class Composicao { +int codigo_composto PK, FK +int codigo_componente PK, FK +int quantidade_na_montagem } Produto "1" -- "*" Composicao : assume papel de 'Todo' Produto "1" -- "*" Composicao : assume papel de 'Peça'

Chave Primária Composta: A PK da tabela Composicao é formada pela união das PKs que vêm da mesma entidade Produto, mas com nomes/papéis diferentes.

5. Lógica do Self-Join (Consulta Prática)

Para recuperar dados de um autorrelacionamento no SQL, o banco de dados trata a mesma tabela como se fossem duas instâncias separadas (A e B) através do uso de Aliases.

graph LR subgraph "SGBD (Memória Virtual)" T1["Tabela A (Papel: Supervisor)"] T2["Tabela B (Papel: Subordinado)"] end T1 -- "Join: A.PK = B.FK" --> T2 T2 -- "Resultado" --> R["Relatório: 'Nome Chefe' | 'Nome Subordinado'"]

Funcionamento: A cláusula FROM Funcionário A, Funcionário B permite que o sistema compare o nss da Tabela A com o nss_supervisor da Tabela B.

Como aplicar na Prática

  • Gestão de Recursos Humanos (Organogramas): Esta é a aplicação mais clássica do autorrelacionamento 1:N. Em sistemas de RH, utiliza-se uma única tabela para todos os colaboradores, onde uma coluna de Chave Estrangeira (FK) aponta para a Chave Primária (PK) do próprio colega que exerce a função de gestor.
    • Na prática: Ao cadastrar um vendedor, o sistema associa o campo Chefe_FK ao código de outro vendedor que já está na base. Isso permite reconstruir toda a árvore de comando da empresa através de consultas do tipo Self-Join.
  • Estrutura de Materiais em Indústrias (BOM - Bill of Materials): Em ambientes fabris, utiliza-se o autorrelacionamento N:N para gerenciar a composição de produtos. Como um produto acabado (ex: motor) é feito de vários componentes (ex: pistões) e um componente pode participar de vários produtos diferentes, a relação é Muitos-para-Muitos.
    • Na prática: Cria-se uma entidade associativa (ex: COMPOSIÇÃO) que armazena dois IDs da mesma tabela de PRODUTO: um no papel de "todo" e outro no papel de "peça", além da quantidade necessária para a montagem.
  • Sistemas de Catalogação e Taxonomia (Biologia e Anatomia): Para modelar a decomposição de objetos complexos em partes menores, utiliza-se a navegação recursiva.
    • Na prática: Uma entidade PARTES_DO_CORPO pode representar que a "Cabeça" é composta pelo "Olho", e o "Olho" é composto pela "Retina". No banco de dados, o registro da "Retina" terá como "pai" o ID do "Olho", permitindo que o sistema navegue do topo para as folhas da hierarquia.
  • Hierarquia de Unidades Corporativas (Matriz e Filiais): Empresas com estruturas complexas utilizam o autorrelacionamento para definir a dependência entre sedes e unidades regionais.
    • Na prática: Uma entidade única EMPRESA pode conter um atributo que indica se ela é "sede" ou "filial". Um relacionamento reflexivo permite identificar exatamente qual empresa é a sede de qual filial, simplificando o modelo que antes teria tabelas separadas para o mesmo conjunto de atributos.
  • Categorização de Produtos em E-commerce: Para criar menus de navegação com categorias e subcategorias infinitas (ex: Eletrônicos > Computadores > Notebooks), utiliza-se a recursividade.
    • Na prática: Cada categoria possui uma referência para sua categoria "pai" dentro da mesma tabela. Se o campo estiver Nulo, o sistema entende que aquela é uma categoria raiz (nível mais alto).
  • Sistemas de Permissões e Segurança (Acesso Hierárquico): Em grandes corporações, os níveis de acesso a dados seguem a hierarquia de visões dos departamentos.
    • Na prática: As permissões são herdadas; se um gerente tem acesso a uma pasta "pai", ele automaticamente pode visualizar as subpastas "filhas" através da lógica de navegação hierárquica implementada no banco.
