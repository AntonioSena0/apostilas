Teoria & Conceitos

A Álgebra Relacional é uma linguagem de consulta procedural, o que significa que ela descreve o "como" os dados devem ser recuperados através de uma sequência de operações passo a passo. Ela serve como o fundamento teórico para o modelo relacional e para a linguagem SQL, permitindo manipular relações (tabelas) para gerar novos conjuntos de dados como resposta.

Origem e Fundamentação Matemática

A Álgebra Relacional não nasceu como uma ferramenta de programação, mas como uma resposta à necessidade de injetar rigor científico e princípios sólidos no campo do gerenciamento de dados.

  • O Criador: Foi formulada originalmente por Edgar F. Codd (Ted Codd), um pesquisador da IBM, em seu artigo revolucionário de 1970, intitulado "A Relational Model of Data for Large Shared Data Banks".
  • Base Teórica: Codd, matemático de formação, utilizou a Teoria dos Conjuntos e a Lógica de Predicados para descrever como os dados deveriam ser estruturados e recuperados de forma transparente ao usuário.
  • Evolução: Em 1979, Codd e Chris Date refinaram essas operações, dando origem ao que hoje conhecemos como o modelo relacional estendido.
  • A Filosofia de Codd: Ele defendia a simplicidade e a unicidade. Como ele mesmo dizia: "Se algo é verdadeiro, dizer isso duas vezes não o torna mais verdadeiro" — justificando por que a álgebra relacional proíbe tuplas duplicadas e exige chaves primárias.

1. Operação de Seleção (σ)

A seleção é um operador unário — ou seja, atua sobre apenas uma tabela por vez — e é utilizada para filtrar as linhas (tuplas) que atendem a um critério específico.

  • Função: Atua como um filtro horizontal. Ela seleciona um subconjunto de linhas que satisfazem um predicado lógico (como igualdade, maior ou menor que), mantendo todas as colunas originais da tabela.
  • Esquema Resultante: A relação resultante tem exatamente o mesmo esquema (conjunto de colunas) que a relação de entrada.
  • Exemplo Prático: Se você deseja buscar apenas os funcionários do sexo 'M', a seleção percorre a tabela e descarta todas as linhas que não atendem a essa condição.

2. Operação de Projeção (π)

A projeção também é um operador unário, mas foca na filtragem das colunas (atributos) de uma relação, descartando as que não foram solicitadas.

  • Função: Atua como um filtro vertical. Ela cria uma nova tabela contendo apenas as colunas especificadas na lista de argumentos.
  • Eliminação de Duplicidade: Um conceito fundamental da álgebra relacional é que o resultado deve ser um conjunto matemático puro. Por isso, a projeção remove automaticamente linhas duplicadas que possam surgir após a exclusão de colunas identificadoras (como chaves primárias).
  • Grau da Relação: O "grau" (número de colunas) da tabela resultante será igual ao número de atributos listados na operação.

3. Produto Cartesiano (×)

Diferente dos anteriores, o produto cartesiano é um operador binário, pois atua sobre duas tabelas distintas para combinar suas informações.

  • Função: É a forma primitiva de unir tabelas heterogêneas (de assuntos diferentes). Ele combina cada linha da primeira tabela com todas as linhas da segunda tabela.
  • Cálculo de Dimensões:
    • Atributos: Se a tabela A tem n colunas e a tabela B tem m colunas, o resultado terá n+m colunas.
    • Tuplas: Se a tabela A tem 10 linhas e a tabela B tem 5 linhas, o resultado terá 10×5=50 linhas.
  • Utilidade: Raramente é usado sozinho para exibição de dados, servindo como a base para operações de Junção (Join), onde o resultado massivo do produto é filtrado por uma Seleção para manter apenas as combinações logicamente relacionadas (como ID_DEPARTAMENTO igual em ambas as tabelas).

Comparação: Teoria (Álgebra) vs. Uso Prático (SQL)

Embora o SQL seja a linguagem utilizada no dia a dia, a álgebra relacional é o motor que define como o SGBD processa internamente os dados. Compreender essa diferença é o que separa um desenvolvedor comum de um especialista em banco de dados.

1. Procedural vs. Declarativo

  • Álgebra Relacional (Procedural): Descreve o "como" os dados devem ser recuperados através de uma sequência de passos lógicos. Você precisa definir a ordem exata das operações (ex: primeiro filtrar as linhas, depois isolar as colunas).
  • SQL (Declarativo): Descreve o "o quê" você deseja obter, deixando para o otimizador do banco de dados a tarefa de decidir a melhor estratégia interna de execução.

2. Mapeamento de Comandos

Os operadores fundamentais da álgebra têm equivalentes diretos nas cláusulas SQL que utilizamos rotineiramente, como SELECT (Projeção), WHERE (Seleção) e FROM (Produto Cartesiano).

3. A Propriedade do Fechamento (Closure)

Este é um dos conceitos mais importantes para o uso prático. Tanto na álgebra quanto no SQL, o resultado de uma operação sobre uma relação (tabela) é sempre outra relação.

  • Na Prática: É essa propriedade que permite a criação de subconsultas (subqueries) e o aninhamento de comandos SQL complexos. Você pode usar o resultado de um SELECT como se fosse uma tabela real dentro de outro comando.

4. Importância para a Performance

Entender álgebra relacional permite que o analista compreenda a navegação interna do banco de dados. Por exemplo, saber que a Seleção (σ) reduz o número de linhas antes de uma junção cara ajuda a entender por que filtros no WHERE são fundamentais para que a consulta não trave o sistema em grandes massas de dados.

Exemplo de Código

Abaixo, apresentamos a comparação prática entre os comandos SQL e seus respectivos operadores na Álgebra Relacional.

1. Operação de Projeção (π)

A projeção é um operador unário que realiza um filtro vertical, selecionando apenas colunas específicas.

-- SQL: Seleciona apenas as colunas de nome e cargo
SELECT nome_vendedor, faixa_de_comissao 
FROM vendedores;

-- ÁLGEBRA RELACIONAL: π nome_vendedor, faixa_de_comissao (VENDEDORES)
-- Comentário: O SELECT do SQL equivale à Projeção (π). 
-- Na teoria, se houver linhas duplicadas após a projeção, elas são removidas 
-- automaticamente para manter a propriedade de conjunto.

2. Operação de Seleção (σ)

A seleção é um operador unário que realiza um filtro horizontal, selecionando linhas que atendem a um predicado.

-- SQL: Filtra apenas os produtos com valor superior a 2.00
SELECT * 
FROM produto 
WHERE valor_unitario > 2.00;

-- ÁLGEBRA RELACIONAL: σ valor_unitario > 2.00 (PRODUTO)
-- Comentário: A cláusula WHERE do SQL equivale à Seleção (σ). 
-- O resultado mantém todas as colunas originais (o esquema não muda), 
-- mas reduz o número de tuplas (linhas).

3. Projeção e Seleção Combinadas

Na prática, quase sempre combinamos os dois operadores para filtrar colunas e linhas simultaneamente.

-- SQL: Lista o nome e endereço apenas de clientes de 'Niterói'
SELECT nome_cliente, endereco 
FROM cliente 
WHERE cidade = 'Niteroi';

-- ÁLGEBRA RELACIONAL: π nome_cliente, endereco (σ cidade = 'Niteroi' (CLIENTE))
-- Comentário: A execução é procedural (passo a passo): primeiro o SGBD 
-- aplica a Seleção (σ) para reduzir as linhas e, sobre esse resultado, 
-- aplica a Projeção (π) para isolar as colunas.

4. Produto Cartesiano (×)

Operador binário que combina todas as tuplas de uma tabela com todas as tuplas de outra.

-- SQL: Combina cada linha de funcionários com cada linha de cargos
SELECT * 
FROM funcionario, cargo;
-- Ou explicitamente: SELECT * FROM funcionario CROSS JOIN cargo;

-- ÁLGEBRA RELACIONAL: FUNCIONARIO × CARGO
-- Comentário: Equivale à cláusula FROM com múltiplas tabelas. 
-- Se a Tabela A tem 10 linhas e a B tem 5, o resultado terá 50 linhas. 
-- Raramente usado sozinho sem um filtro de junção posterior.

5. Junção (⋈)

A junção é, teoricamente, uma Seleção aplicada sobre um Produto Cartesiano.

-- SQL: Une tabelas baseando-se em uma coluna comum (chave estrangeira)
SELECT nome_cliente, numero_pedido 
FROM cliente 
INNER JOIN pedido ON cliente.codigo_cliente = pedido.codigo_cliente;

-- ÁLGEBRA RELACIONAL: π nome_cliente, numero_pedido (CLIENTE ⋈ codigo_cliente = codigo_cliente PEDIDO)
-- Comentário: O operador de Junção (⋈) resolve o produto cartesiano filtrando 
-- apenas onde os valores das chaves coincidem, efetivando o relacionamento lógico.

6. Operação de União (∪)

Operador binário que consolida os registros de dois conjuntos compatíveis.

-- SQL: Lista clientes do RJ e vendedores com altos salários em uma só lista
SELECT nome_cliente FROM cliente WHERE UF = 'RJ'
UNION
SELECT nome_vendedor FROM vendedores WHERE salario_fixo > 5000.00;

-- ÁLGEBRA RELACIONAL: (π nome_cliente (σ UF='RJ' (CLIENTE))) ∪ (π nome_vendedor (σ salario>5000 (VENDEDORES)))
-- Comentário: Para a União (∪) funcionar, as tabelas devem ser "união-compatíveis": 
-- mesmo número de colunas e domínios de dados idênticos.

7. Diferença de Conjuntos (−)

Retorna elementos que existem no primeiro conjunto mas não existem no segundo.

-- SQL: Clientes que possuem CONTA mas não possuem EMPRÉSTIMO
SELECT cod_cliente FROM conta_corrente
EXCEPT
SELECT cod_cliente FROM emprestimo;
-- No MySQL, usa-se subqueries com NOT IN para simular a Diferença.

-- ÁLGEBRA RELACIONAL: π cod_cliente (CONTA_CORRENTE) − π cod_cliente (EMPRESTIMO)
-- Comentário: O operador de diferença (−) remove do resultado as tuplas 
-- que são comuns a ambos os conjuntos.

8. Interseção (∩)

Retorna apenas as tuplas que existem simultaneamente em ambos os conjuntos.

-- SQL: Clientes que possuem simultaneamente CONTA e EMPRÉSTIMO
SELECT cod_cliente FROM conta_corrente
INTERSECT
SELECT cod_cliente FROM emprestimo;

-- ÁLGEBRA RELACIONAL: π cod_cliente (CONTA_CORRENTE) ∩ π cod_cliente (EMPRESTIMO)
-- Comentário: Pode ser expressa através de uniões e diferenças, mas o operador 
-- de Interseção (∩) facilita a busca por dados comuns entre tabelas compatíveis.

Resumo de Mapeamento Técnico

Comando SQL Operador Álgebra Tipo de Filtro
SELECT Projeção (π) Vertical (Colunas)
WHERE Seleção (σ) Horizontal (Linhas)
FROM (A, B) Produto (×) Combinação de Conjuntos
JOIN / ON Junção (⋈) Combinação Relacional
UNION União (∪) Consolidação de Conjuntos
AS Renomear (ρ) Identificação de Resultados

Dicas & Lembretes

Conceitos Fundamentais
  • A Álgebra é Procedural: Diferente do SQL (que é declarativo), a álgebra relacional descreve o "como" os dados devem ser recuperados através de uma sequência de operações lógicas.
  • Propriedade do Fechamento (Closure): Este é o conceito mais importante. O resultado de qualquer operação de álgebra é sempre uma nova relação (tabela). Isso é o que permite o aninhamento de expressões (uma operação dentro da outra).
  • Eliminação de Duplicidade: Na álgebra relacional pura (teoria dos conjuntos), não existem linhas duplicadas. Ao realizar uma Projeção (π), o sistema remove automaticamente tuplas repetidas para garantir que o resultado seja um conjunto válido.
Regras de Operação
  • Compatibilidade de União: Para utilizar os operadores de União (∪), Interseção (∩) e Diferença (−), as tabelas devem ser "união-compatíveis": precisam ter o mesmo número de colunas e os domínios (tipos) correspondentes devem ser idênticos.
  • Precedência e Parênteses: Em expressões complexas, as operações dentro dos parênteses são executadas primeiro (lógica bottom-up).
  • Produto vs. Junção: O Produto Cartesiano (×) combina todas as linhas de uma tabela com todas as linhas de outra, gerando resultados massivos. A Junção (⋈) é mais eficiente, pois equivale a um Produto seguido de uma Seleção que filtra apenas os registros relacionados.
Dicas de Performance e Prática
  • Filtre o Quanto Antes: Uma regra de ouro da otimização é aplicar a Seleção (σ) o mais cedo possível. Filtrar as linhas antes de realizar um Produto Cartesiano ou Junção reduz drasticamente o volume de dados processados pelo SGBD.
  • Cuidado com a Diferença: A operação de Diferença (−) não é comutativa. A ordem dos fatores altera o resultado: A−B (linhas que estão em A mas não em B) é diferente de B−A.
  • Use Nomes Intermediários: Para facilitar a leitura de consultas longas, utilize o operador de Atribuição (←) ou Renomeação (ρ) para salvar resultados parciais em "tabelas temporárias".
  • Agregações geram Relações: Lembre-se de que o resultado de funções como SOMA ou CONTADOR ainda é uma relação (uma tabela com uma linha e uma coluna), e não apenas um valor numérico solto.
Lembrete de Mapeamento (Álgebra → SQL)
  • σ (Seleção) ≈ Cláusula WHERE.
  • π (Projeção) ≈ Cláusula SELECT.
  • × (Produto) ≈ Cláusula FROM com múltiplas tabelas.
  • ρ (Renomear) ≈ Cláusula AS.

Como aplicar na Prática

  • Planejamento Lógico de Consultas: Utilize a natureza procedural da álgebra relacional para desenhar o "caminho" que os dados devem percorrer. Antes de escrever um comando complexo, esboce a sequência de operações: primeiro identifique quais tabelas serão combinadas (Produto), quais filtros de linha são necessários (Seleção) e, por fim, quais colunas devem ser exibidas (Projeção).
  • Otimização Preventiva de Performance: Aplique a regra de ouro da otimização: filtrem as linhas o quanto antes. Na prática, isso significa entender que realizar uma Seleção (σ) antes de um Produto Cartesiano (×) reduz drasticamente o volume de dados processados pelo SGBD, evitando a criação de tabelas intermediárias imensas e desnecessárias.
  • Análise de Execução Interna (Query Tuning): Use o conhecimento da álgebra para interpretar como o SGBD processa os comandos internamente. Ao entender que todo comando SQL é desmembrado em operações algébricas, o analista consegue visualizar melhor a utilização de índices em filtros de seleção e a eficiência das junções em consultas que envolvem múltiplas tabelas.
  • Resolução de Ambiguidades em Auto-relacionamentos: Em situações onde uma consulta precisa comparar dados de uma tabela com ela mesma (self-join), aplique o conceito do operador Renomear (ρ). Isso permite criar "apelidos" lógicos para a mesma relação, garantindo que o produto cartesiano seja executado sem erros de referência duplicada.
  • Validação de Resultados: Utilize a propriedade do fechamento (closure) para criar subconsultas ou visões intermediárias. Como o resultado de uma operação algébrica é sempre uma nova relação, você pode testar partes isoladas de uma consulta complexa (como apenas a seleção de um subconjunto) antes de integrá-la ao produto final.
