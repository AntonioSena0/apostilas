Teoria & Conceitos

Nesta aula, exploraremos como proteger suas aplicações web tratando os dados que vêm do mundo externo. Aprenderemos a distinguir entre limpar dados e verificar se eles cumprem regras específicas, utilizando ferramentas nativas e modernas do PHP.

1.1. O Princípio da Desconfiança

O pilar fundamental da segurança web é: Nunca confie em dados vindos do usuário ou de fontes externas. Isso inclui formulários HTML, parâmetros de URL (query strings), cookies e até cabeçalhos HTTP. Esses dados podem ser manipulados para injetar scripts maliciosos ou burlar a lógica do sistema.

1.2. Sanitização vs. Validação

Embora usados juntos, são processos distintos:

  • Sanitização (Limpeza): É o processo de remover caracteres indesejados ou perigosos de uma entrada. O objetivo é "limpar" o dado para que ele possa ser processado ou exibido com segurança, sem alterar necessariamente sua essência (ex: remover tags HTML de um comentário).
  • Validação (Verificação): É o processo de verificar se o dado atende a critérios pré-definidos. O objetivo é confirmar se o dado é o que se espera que seja (ex: verificar se um campo de "idade" contém apenas números inteiros entre 0 e 120).

1.3. O Uso de filter_input()

A função filter_input() é a maneira recomendada de obter variáveis externas em PHP. Ela combina a obtenção do dado com a filtragem, evitando o acesso direto a superglobais como $_GET ou $_POST, o que é uma prática de segurança mais robusta.

Sintaxe: filter_input(tipo_origem, nome_variavel, filtro, opcoes)

  • Tipos de Origem: INPUT_GET, INPUT_POST, INPUT_COOKIE, INPUT_SERVER, INPUT_ENV.
  • Filtros de Sanitização: Ex: FILTER_SANITIZE_EMAIL, FILTER_SANITIZE_SPECIAL_CHARS.
  • Filtros de Validação: Ex: FILTER_VALIDATE_INT, FILTER_VALIDATE_EMAIL, FILTER_VALIDATE_URL.

Exemplo de Sanitização:

// Remove tags e caracteres especiais de uma string vinda do POST
$nome = filter_input(INPUT_POST, 'nome', FILTER_SANITIZE_SPECIAL_CHARS);

echo $nome; // Se o usuário enviou "<h1>Olá</h1>", resultará em "&#60;h1&#62;Olá&#60;/h1&#62;"

Por que funciona assim?

Quando você aplica a sanitização de caracteres especiais:

  • O dado original: <h1>Olá</h1>
  • A ação do filtro: Ele identifica caracteres que têm significado especial em HTML (como < e >) e os converte em suas respectivas Entidades HTML.
  • A conversão:
    • < vira &#60; (ou &lt;)
    • > vira &#62; (ou &gt;)
  • O resultado no código-fonte: &#60;h1&#62;Olá&#60;/h1&#62;

Qual o benefício disso? (Segurança)

Ao fazer essa conversão, o navegador não interpretará mais o texto como uma tag de título (<h1>). Em vez de deixar o texto grande e em negrito, o navegador exibirá literalmente os caracteres <h1>Olá</h1> na tela do usuário.

Isso impede ataques de XSS (Cross-Site Scripting), pois se um usuário mal-intencionado enviasse um script como:
<script>alert('Ataque!')</script>

O PHP converteria para:
&#60;script&#62;alert('Ataque!')&#60;/script&#62;

E o script não seria executado pelo navegador.

Lembrete importante: Se você usar o filtro FILTER_SANITIZE_STRING (descontinuado no PHP 8.1), o comportamento seria remover as tags completamente, restando apenas "Olá". Por isso, o uso de FILTER_SANITIZE_SPECIAL_CHARS ou a função htmlspecialchars() é o padrão atual e mais seguro.

Exemplo de Validação:

// Verifica se a idade é um inteiro entre 18 e 100
$idade = filter_input(INPUT_POST, 'idade', FILTER_VALIDATE_INT, [
    "options" => [
        "min_range" => 18, 
        "max_range" => 100
    ]
]);

if ($idade === false) {
    echo "Idade inválida ou fora do intervalo permitido.";
}

1.4. Proteção contra Manipulação de Tipos (Type Juggling)

O PHP é uma linguagem de tipagem fraca e dinâmica, o que permite "conversões automáticas" de tipos que podem levar a falhas de segurança se não forem tratadas.

O Problema: Comparações Flexíveis (==)

Muitas vezes, desenvolvedores usam == para comparar dados, mas o PHP pode converter strings em números de forma inesperada. O uso de === ainda é a regra de ouro para evitar comportamentos imprevisíveis.

A Solução: Comparações Estritas e Tipagem no PHP 8

  • Operadores de Identidade (===): Sempre use === para comparar valor e tipo, garantindo que "0" (string) não seja igual a 0 (inteiro).
  • Strict Types: No PHP 8, você pode forçar a tipagem estrita no início do arquivo. Isso impede que uma função que espera um int aceite uma string numérica silenciosamente.

Exemplo de Proteção com Tipagem Estrita:

declare(strict_types=1);

function processarIdade(int $idade) {
    return "A idade é $idade";
}

// Se o dado vier como string do formulário, o PHP lançará um erro fatal em vez de tentar converter
try {
    echo processarIdade($_POST['idade']); // Se $_POST['idade'] for "25", causará erro com strict_types
} catch (TypeError $e) {
    echo "Erro: O sistema esperava um número inteiro.";
}

1.5. Boas Práticas no Fluxo de Dados

Para uma segurança completa, siga este fluxo:

  1. Entrada: Use filter_input() para sanitizar e validar assim que o dado entrar no script.
  2. Processamento: Trabalhe com variáveis validadas e tipos estritos (PHP 8).
  3. Saída (Escaping): Ao exibir dados de volta no HTML, use funções como htmlspecialchars() para impedir ataques de Cross-Site Scripting (XSS).

Exemplo de Código

1. Uso do filter_input para Sanitização

A sanitização limpa os dados removendo ou convertendo caracteres perigosos.

// Sanitizando um nome (remove tags HTML)
$nome = filter_input(INPUT_POST, 'nome', FILTER_SANITIZE_SPECIAL_CHARS);

// Sanitizando um e-mail (remove caracteres não permitidos em e-mails)
$email = filter_input(INPUT_GET, 'email', FILTER_SANITIZE_EMAIL);

echo "Nome seguro: " . $nome; // Ex: se enviar <h1>Olá</h1>, resultará em &lt;h1&gt;Olá&lt;/h1&gt;

2. Uso do filter_input para Validação

A validação verifica se o dado segue uma regra específica (ex: se é um e-mail válido).

// Validando um e-mail
$email = filter_input(INPUT_POST, 'email', FILTER_VALIDATE_EMAIL);

if ($email === false) {
    echo "E-mail inválido!";
} else {
    echo "E-mail aceito: " . $email;
}

// Validando um número inteiro dentro de um intervalo
$idade_opcoes = [
    "options" => [
        "min_range" => 18,
        "max_range" => 100
    ]
];

$idade = filter_input(INPUT_POST, 'idade', FILTER_VALIDATE_INT, $idade_opcoes);

if ($idade === false || $idade === null) {
    echo "Idade deve ser um número entre 18 e 100.";
}

3. Proteção contra Manipulação de Tipos (Strict Types)

Para evitar que o PHP converta tipos de forma silenciosa e perigosa, utilize a declaração de tipagem estrita no topo do arquivo.

<?php
declare(strict_types=1); // Força o PHP a respeitar rigorosamente os tipos 

function processarIdade(int $idade) {
    return "A idade processada é: " . $idade;
}

try {
    // Se passar "25" (string), o PHP lançará um erro em vez de converter para 25 (int)
    echo processarIdade(25); 
} catch (TypeError $e) {
    echo "Erro de tipo: " . $e->getMessage();
}

4. Proteção na Exibição (XSS)

Sempre que exibir dados vindos de fora no navegador, use htmlspecialchars para impedir a execução de scripts maliciosos.

$comentario_usuario = "<script>alert('ataque!');</script>";

// Forma segura de exibir no HTML
echo htmlspecialchars($comentario_usuario, ENT_QUOTES, 'UTF-8');
// Resultado: &lt;script&gt;alert('ataque!');&lt;/script&gt;

Dicas & Lembretes

  • Nunca confie em dados externos: Trate toda e qualquer entrada vinda do usuário ou de fontes externas — como formulários HTML, parâmetros de URL (query strings), cookies e cabeçalhos HTTP — como potencialmente maliciosa.
  • Utilize sempre o filter_input(): Evite acessar diretamente superglobais como $_POST ou $_GET. O uso desta função é a prática recomendada para obter dados externos de forma mais segura.
  • Combine Sanitização e Validação: Primeiro, realize a Sanitização para "limpar" o dado (remover caracteres desnecessários ou perigosos) e, em seguida, aplique a Validação para verificar se o resultado final cumpre as regras de negócio ou técnicas exigidas.
  • Ative a Tipagem Estrita: Utilize a declaração declare(strict_types=1) no topo dos seus arquivos PHP. Isso impede que o PHP realize conversões de tipos silenciosas que podem levar a comportamentos inesperados e falhas de segurança.
  • Proteja a Exibição de Dados (XSS): Ao exibir qualquer dado vindo de fora no navegador, utilize sempre a função htmlspecialchars(). Isso converte caracteres especiais em entidades HTML, impedindo que scripts maliciosos sejam executados no browser do usuário.
  • Entenda a diferença técnica:
    • Sanitização: Focada em limpar o que está "sujo" (ex: remover tags HTML de um comentário).
    • Validação: Focada em conferir se a informação é o que diz ser (ex: verificar se um campo de e-mail contém realmente um endereço de e-mail válido).

Como aplicar na Prática

1. Sistema de Comentários ou Fórum (Prevenção de XSS)

Em um projeto real, usuários podem tentar enviar scripts maliciosos em campos de texto.

  • Ação: Ao receber o comentário, utilize FILTER_SANITIZE_SPECIAL_CHARS para converter caracteres como < e > em entidades HTML.
  • Exibição: Antes de imprimir no navegador, reforce com htmlspecialchars().
  • Resultado: O navegador exibe o código do script como texto puro em vez de executá-lo.

2. Cadastro de Usuários (Integridade de Dados)

Ao criar um formulário de cadastro, você precisa garantir que os campos técnicos (e-mail, idade, site) sejam legítimos.

  • E-mail: Use filter_input(INPUT_POST, 'email', FILTER_VALIDATE_EMAIL). Se retornar false, o cadastro deve ser interrompido com uma mensagem de erro.
  • Idade: Use FILTER_VALIDATE_INT com a opção min_range para garantir que apenas maiores de 18 anos se cadastrem.
  • URL do Perfil: Use FILTER_VALIDATE_URL para garantir que o link enviado seja um endereço web válido.

3. Painel Administrativo com Busca (Proteção de URL)

Muitos sistemas utilizam parâmetros via GET para filtrar resultados (ex: lista.php?status=ativo).

  • Ação: Um atacante pode mudar a URL para ?status=123 tentando burlar a lógica.
  • Prática: Sempre sanitize o parâmetro da URL com FILTER_SANITIZE_FULL_SPECIAL_CHARS e valide se o valor recebido pertence a uma lista de opções permitidas (whitelist).

4. Integração com Banco de Dados (Prevenção de Injeção)

Embora a sanitização ajude, ela não substitui o uso de Prepared Statements (Instruções Preparadas) no banco de dados.

  • Fluxo Real:
    1. O dado chega (ex: $_POST['id']).
    2. Você valida se é um inteiro com filter_input.
    3. Você envia o dado validado para o banco de dados usando PDO com bindParam.

5. Cálculo de Frete ou E-commerce (Proteção de Tipos)

Em sistemas que lidam com cálculos financeiros, a manipulação de tipos (ex: tratar uma string como número) pode gerar erros graves.

  • Prática: Use declare(strict_types=1) no topo dos arquivos de lógica de negócio.
  • Benefício: Se uma função espera um float (preço) e recebe acidentalmente uma string, o PHP interrompe a execução imediatamente, evitando que um cálculo errado seja processado e salvo no banco.
