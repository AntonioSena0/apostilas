Teoria & Conceitos

1. A Filosofia do Mapeamento Objeto-Relacional (ORM)

No desenvolvimento de sistemas multicamadas, o Eloquent ORM atua como uma ponte entre o Modelo Conceitual (objetos em PHP) e o Modelo Físico (tabelas no banco de dados). Em vez de manipular linhas e colunas através de SQL puro, você interage com classes que possuem "identidades e responsabilidades".

A grande vantagem de definir relacionamentos no Laravel é que eles são declarados como métodos. Isso permite que o relacionamento funcione como um Query Builder, possibilitando o encadeamento de filtros antes da execução da consulta. Por exemplo: $usuario->pedidos()->where('status', 'pago')->get();.

2. Relacionamento Um-para-Um (1:1)

Este é o tipo mais básico de associação, onde um registro em uma tabela está ligado a exatamente um registro em outra.

  • Conceito: Frequentemente usado para estender dados de um modelo sem sobrecarregar a tabela principal (ex: um Usuario e seu PerfilSettings).
  • Implementação: Usa-se o método hasOne() no modelo pai e belongsTo() no modelo filho.
  • Convenção de Chave Estrangeira: O Laravel busca automaticamente por uma coluna chamada model_id (ex: user_id) na tabela do modelo relacionado.

Exemplo de Código:

// Modelo User (Pai)
public function phone(): HasOne {
    return $this->hasOne(Phone::class); // Um usuário tem um telefone
}

// Modelo Phone (Filho)
public function user(): BelongsTo {
    return $this->belongsTo(User::class); // O telefone pertence a um usuário
}

3. Relacionamento Um-para-Muitos (1:N)

É o relacionamento mais comum em aplicações web, onde um único modelo pai possui um ou mais modelos filhos.

  • Conceito: Um pai (como um Post) pode ter um número infinito de filhos (como Comments).
  • Persistência Complexa: Para vincular um filho a um pai, você pode usar o método associate() no modelo filho (que define a chave estrangeira na memória) ou o método save() a partir do pai.

Exemplo de Código:

// Modelo Post
public function comments(): HasMany {
    return $this->hasMany(Comment::class); // Um post tem muitos comentários
}

// Lógica de Persistência com createMany (Laravel 5.4+)
$post = Post::find(1);
$post->comments()->createMany([
    ['body' => 'Dica excelente!'],
    ['body' => 'Laravel 13 é muito produtivo.']
]); // Inserção em lote garantindo o post_id correto

4. Relacionamento Muitos-para-Muitos (N:N)

Este é o cenário mais complexo, onde múltiplos registros em uma tabela estão ligados a múltiplos registros em outra.

  • O Papel da Tabela Pivô: Como as tabelas relacionais não podem armazenar listas de IDs diretamente, utiliza-se uma tabela intermediária (tabela pivô).
  • Convenção de Nomenclatura: Por padrão, o nome da tabela pivô deve ser o nome dos dois modelos no singular, em ordem alfabética, separados por underline (ex: role_user para os modelos Role e User).
  • Atributos de Pivô: Tabelas pivô podem conter dados extras (como created_at ou status) que podem ser acessados via atributo pivot ou renomeados usando o método as().

Exemplo de Código:

// Modelo User
public function roles(): BelongsToMany {
    return $this->belongsToMany(Role::class)
                ->as('membership') // Nome personalizado para o objeto de pivô
                ->withPivot('active') // Permite acessar colunas extras na pivô
                ->withTimestamps(); // Mantém campos created_at/updated_at na pivô
}

// Manipulação Automática com sync()
$user->roles()->sync([28, 29, 30]); 
// O sync() é a forma mais poderosa de persistência N:N: ele remove o que não está 
// na lista, mantém o que já existia e adiciona apenas o novo.

5. Otimização e Segurança: Eager Loading (Problema N+1)

Um erro crítico em conceitos de ORM é carregar dados relacionados dentro de um loop, o que gera o problema de performance "N+1".

  • O Problema: Se você listar 25 contatos e buscar os telefones de cada um dentro do loop, o sistema fará 1 consulta para os contatos + 25 consultas individuais para os telefones.
  • A Solução: Use o método with() para carregar preventivamente os dados em apenas 2 consultas totais.

Exemplo:

$contatos = Contact::with('phoneNumbers')->get(); 
// Carregamento ansioso (Eager Loading) - apenas 2 consultas totais

6. Persistência de Integridade (Foreign Keys)

Ao projetar bancos de dados, garantir a integridade referencial é fundamental. O Laravel facilita isso através das Migrations, onde você define que uma chave estrangeira deve ser removida em cascata caso o pai seja deletado:

$table->foreignId('user_id')->constrained()->onDelete('cascade');

Isso garante que quando um usuário for deletado, todos os seus registros relacionados também sejam, mantendo a consistência dos dados.

Exemplo de Código

Abaixo estão os cenários fundamentais, utilizando a sintaxe moderna de tipos do PHP 8.x integrada ao Laravel.

Cenário 1: Um-para-Um (1:1) – Usuário e Perfil

// app/Models/User.php
public function profile(): HasOne {
    return $this->hasOne(Profile::class); // Define que um usuário possui um perfil
}

// app/Models/Profile.php
public function user(): BelongsTo {
    return $this->belongsTo(User::class); // Inverso do relacionamento
}

Cenário 2: Um-para-Muitos (1:N) – Post e Comentários

// app/Models/Post.php
public function comments(): HasMany {
    return $this->hasMany(Comment::class); // Um post pode ter múltiplos comentários
}

// Persistência Complexa: Criando vários comentários de uma vez
$post = Post::find(1);
$post->comments()->createMany([
    ['body' => 'Excelente post!'],
    ['body' => 'Laravel 13 facilitou muito a vida.'],
]);

Cenário 3: Muitos-para-Muitos (N:N) – Usuários e Papéis (Roles)

// app/Models/User.php
public function roles(): BelongsToMany {
    return $this->belongsToMany(Role::class)->withTimestamps(); // Uso de tabela pivô
}

// Lógica de Persistência com sync()
// O sync() remove o que não está no array e adiciona o que é novo, garantindo integridade
$user->roles()->sync([31-33]);

Cenário 4: Vinculação (Associate) em 1:N

Para alterar o "pai" de um objeto sem deletá-lo:

$comment = Comment::find(10);
$comment->post()->associate(Post::find(5)); // Define o novo post pai [30, 34, 35]
$comment->save();

Dicas & Lembretes

  • Evite o Problema N+1: Sempre que for listar modelos com seus filhos, use with() (Eager Loading). Em vez de 26 consultas para 25 livros, o Laravel fará apenas 2.
  • Sintaxe de Atributos: No Laravel 12/13, você pode usar PHP Attributes para configurar o comportamento da model diretamente acima da declaração da classe, como o atributo #[Fillable].
  • Propriedades Dinâmicas vs Métodos: Acesse como propriedade ($user->posts) para obter o resultado (Coleção); acesse como método ($user->posts()) para obter a instância da consulta e continuar filtrando.
  • Segurança (Mass Assignment): Lembre-se de definir $fillable ou $guarded na sua Model antes de usar o método create() para evitar vulnerabilidades.

Como aplicar na Prática

O domínio de relacionamentos Eloquent é o que permite construir os sistemas multicamadas exigidos pelo mercado:

  • E-commerce: Relacionar Order (Pedido) com muitos Products através de uma tabela pivô order_items, armazenando preço e quantidade na pivô.
  • Sistemas de Permissão: Controlar o acesso a rotas verificando se o usuário possui a Role (Papel) necessária no banco de dados.
  • Blogs e Redes Sociais: Implementar sistemas de tags (N:N) onde um post tem muitas tags e uma tag pertence a muitos posts.
  • Gestão de Inventário: Modelar peças de automóveis (como no sistema de Bob's Auto Parts) vinculadas a fornecedores e categorias.
  • Aplicações Escolares: Vincular Alunos a Disciplinas e Professores, gerenciando notas na tabela de relacionamento.
