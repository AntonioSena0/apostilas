Teoria & Conceitos: Validação Profissional com Laravel

1. O Princípio da Desconfiança

O pilar fundamental da segurança em sistemas multicamadas é: nunca confie em dados vindos de fontes externas. No Laravel, a validação não é apenas uma checagem de erros, mas um mecanismo de defesa contra ataques e uma garantia de integridade para o banco de dados. Toda entrada — seja via formulários HTML, parâmetros de URL, cabeçalhos HTTP ou payloads JSON — deve ser tratada como potencialmente maliciosa.

2. O Método validate no Controller

A abordagem mais comum para validar dados no Laravel é utilizar o método validate disponível diretamente no objeto de requisição (Request).

  • Fluxo Automático: Se as regras passarem, o código continua normalmente. Se falharem, o Laravel lança uma exceção (ValidationException) que gera automaticamente uma resposta de erro.
  • Comportamento Inteligente: Em requisições tradicionais de navegador, o Laravel redireciona o usuário de volta para a página anterior com os erros gravados na sessão. Em requisições AJAX/API, ele retorna um JSON com status 422 Unprocessable Entity.

Exemplo de Código (Controller):

public function store(Request $request)
{
    // O Laravel valida os dados e já os retorna filtrados para a variável $validated
    $validated = $request->validate([
        'titulo' => 'required|unique:posts|max:255', // Pipe syntax
        'corpo'  => ['required', 'min:10'],           // Array syntax (mais legível)
    ]);

    // Se chegar aqui, os dados são válidos
    Post::create($validated);
    
    return redirect('/posts');
}

3. Regras de Validação e Sintaxe

As regras podem ser definidas por strings separadas por "pipe" (|) ou em arrays, o que é recomendado para regras complexas ou personalizadas.

Principais Regras de Mercado:

  • required: O campo deve estar presente e não ser vazio.
  • unique:table,column: Verifica se o valor já existe no banco de dados para evitar duplicatas.
  • numeric: Garante que o valor seja um número. Nas versões modernas, você pode usar numeric:strict para garantir que o tipo seja estritamente um inteiro ou float, rejeitando strings numéricas.
  • email: Valida se o formato do e-mail é legítimo.
  • confirmed: Exige que haja um campo correspondente com o sufixo _confirmation (ex: password e password_confirmation).

4. Form Requests: Organização em Larga Escala

Para aplicações complexas, manter as regras dentro do Controller viola o princípio da responsabilidade única (SRP). O Laravel resolve isso com os Form Requests, que são classes dedicadas que encapsulam a lógica de validação e autorização.

Exemplo de Código (Form Request):

// Criado via: php artisan make:request StorePostRequest

public function authorize(): bool
{
    return true; // Define se o usuário tem permissão para esta ação
}

public function rules(): array
{
    return [
        'titulo' => 'required|max:100',
        'slug'   => 'required|alpha_dash|unique:posts',
    ];
}

No Controller, você apenas substitui a classe Request pela sua nova classe:

public function store(StorePostRequest $request) 
{
    // A validação já ocorreu antes de entrar no método!
    $data = $request->validated();
    Post::create($data);
}

5. Exibição de Erros no Front-end (Blade)

O Laravel compartilha automaticamente uma variável $errors com todas as suas views. Você pode usar a diretiva @error para destacar campos específicos.

Exemplo de Código (Blade):

<label for="titulo">Título do Post</label>
<input type="text" name="titulo" class="@error('titulo') is-invalid @enderror">

@error('titulo')
    <div class="alert alert-danger">{% raw %}{{ $message }}{% endraw %}</div>
@enderror

6. Preparação e Sanitização Pré-Validação

Muitas vezes, os dados precisam ser limpos antes de serem validados (ex: remover espaços extras ou formatar um CPF). Dentro de um Form Request, você pode usar o método prepareForValidation.

Exemplo de Código (Sanitização):

protected function prepareForValidation(): void
{
    $this->merge([
        'slug' => str($this->titulo)->slug(), // Gera o slug a partir do título antes da checagem unique
    ]);
}

Exemplos de Código

1. Validação Básica no Controller

Esta é a abordagem inicial, onde as regras são definidas diretamente no método do controlador. É ideal para formulários simples e prototipagem rápida.

public function store(Request $request)
{
    // O método validate redireciona automaticamente em caso de falha [2]
    $validated = $request->validate([
        'titulo' => 'required|unique:posts|max:255', // Regras por string (Pipe syntax) [2]
        'email'  => ['required', 'email'],           // Regras por array (mais legível) [3]
        'status' => 'required|in:rascunho,publicado', // Validação contra lista específica [4]
    ]);

    // Se o código chegar aqui, os dados já estão seguros [2]
    Post::create($validated);
}

2. Validação Avançada: A Regra bail e Atributos Aninhados

Útil quando você quer interromper a verificação de um campo logo na primeira falha ou validar dados vindos de arrays aninhados no HTML.

$request->validate([
    // 'bail' para de validar o campo 'usuario' se 'required' falhar [5]
    'usuario' => 'bail|required|alpha_dash|unique:users',
    
    // Validação de dados aninhados (ex: <input name="contato[telefone]">) [6]
    'contato.telefone' => 'required|numeric',
]);

3. Form Requests (Organização Profissional)

Para aplicações escaláveis, movemos a lógica para uma classe dedicada. Isso segue o princípio da responsabilidade única (SRP) e limpa o controlador.

Gerar classe: php artisan make:request StorePostRequest

// app/Http/Requests/StorePostRequest.php
public function authorize(): bool
{
    return $this->user()->can('create-posts'); // Controle de acesso integrado [8]
}

public function rules(): array
{
    return [
        'titulo' => 'required|min:5|max:100',
        'corpo'  => 'required',
        'imagem' => 'required|image|max:2048', // Validação de arquivo (até 2MB) [4, 9]
    ];
}

4. Preparação e Sanitização (Pre-Validation)

Muitas vezes precisamos limpar os dados (sanitizar) antes que as regras de validação sejam aplicadas. No Laravel 12/13, fazemos isso dentro do Form Request.

protected function prepareForValidation(): void
{
    $this->merge([
        'slug' => str($this->titulo)->slug(),       // Gera slug automaticamente [10]
        'cpf'  => preg_replace('/\D/', '', $this->cpf), // Remove pontos/traços de um CPF
    ]);
}

5. Validação de Arrays e Listas Dinâmicas

Cenário comum em sistemas de e-commerce ou cadastros múltiplos, onde o formulário envia uma lista de itens.

$request->validate([
    'fotos' => 'required|array|min:1',          // Deve ser um array com pelo menos 1 item [12]
    'fotos.*.url' => 'required|url',           // Cada 'url' dentro do array 'fotos' deve ser válida [6, 11]
    'fotos.*.legenda' => 'nullable|string',     // Legenda é opcional [13]
]);

6. Regras Customizadas (Closures)

Para validações muito específicas que não existem no framework, você pode usar uma função anônima.

$request->validate([
    'codigo_promocional' => [
        'required',
        function (string $attribute, mixed $value, Closure $fail) {
            if ($value !== 'ETEC-2024') {
                $fail("O {$attribute} informado é inválido ou expirou."); // Mensagem dinâmica [14]
            }
        },
    ],
]);

7. Integração no Front-end (Blade)

Como exibir os erros de forma amigável para o usuário final.

<!-- Exibir lista geral de erros no topo [16] -->

Dicas & Lembretes

  • Proteção contra Mass Assignment: Use sempre o retorno de $request->validated() ou $request->safe() para preencher seus modelos, garantindo que apenas campos validados entrem no banco.
  • Validação de Arrays: O Laravel permite validar entradas de array usando a notação de ponto (ex: 'fotos.*.url' => 'required|url'), o que é essencial para formulários dinâmicos.
  • TDD (Test Driven Development): Em sistemas de alta qualidade, a validação é o primeiro ponto a ser testado via testes de funcionalidade para garantir que campos obrigatórios nunca fiquem desprotegidos.
  • Validações Mais Utilizadas:
    • required: Campo obrigatório.
    • email: Deve ser um formato de e-mail válido.
    • unique:tabela,coluna: O valor não pode se repetir no banco de dados.
    • confirmed: O campo deve ter um par igual com o sufixo _confirmation (ex: senha).
    • image: O arquivo deve ser uma imagem (jpg, png, etc.).
    • numeric: Garante que a entrada seja um número.
    • nullable: Permite que o campo seja nulo ou vazio.
